# Assumptions

## Overview

The Linux Foundation (LFX) Control-Flow Integrity (CFI) Coding Challenge defines the required functional behavior of the finite state machine but intentionally omits several implementation details. This document records the assumptions made during the RTL design to ensure deterministic behavior and simplify verification.

---

# 1. Synchronous Operation

### Assumption

The FSM operates synchronously using a single positive-edge-triggered clock.

### Justification

This is the standard design methodology for synthesizable RTL and processor control logic.

---

# 2. Asynchronous Reset

### Assumption

The reset signal is asynchronous and immediately initializes the FSM.

After reset:

- Current state = **IDLE**
- Label register = **0x000000**

### Justification

Provides a deterministic startup state and simplifies simulation and verification.

---

# 3. One Packet per Clock Cycle

### Assumption

Exactly one 32-bit packet is presented to the FSM every clock cycle.

### Justification

This behavior is explicitly stated in the coding challenge specification.

---

# 4. Label Programming

### Assumption

The internal label register is updated **only** when:

- Current State = **IDLE**
- Command = **SET**

All other commands leave the label unchanged.

### Justification

This prevents unintended overwriting of the secure label during verification and preserves the integrity of previously programmed values.

---

# 5. JUMP Behavior

### Assumption

A JUMP command only initiates the verification process.

The packet data accompanying JUMP is ignored.

### Justification

According to the challenge description, only the command initiates the transition into the CHECK state.

---

# 6. Landing Pad Verification

### Assumption

Only LPAD commands are considered valid while the FSM is in the CHECK state.

The received label must exactly match the stored label.

### Justification

This models the simplified landing pad verification required by the coding challenge.

---

# 7. Invalid Commands

### Assumption

While in the IDLE state:

- Unknown commands are ignored.
- The FSM remains in IDLE.

While in the CHECK state:

- Any command other than a matching LPAD is treated as a verification failure.

### Justification

This behavior provides deterministic handling of unspecified command values while maintaining secure default behavior.

---


# 8. State Encoding

### Assumption

The FSM uses the default encoding generated by the SystemVerilog enumerated type.

| State | Encoding |
|--------|----------|
| IDLE | 00 |
| CHECK | 01 |
| ERROR | 10 |

### Justification

Using `typedef enum logic [1:0]` improves readability while allowing synthesis tools to generate efficient hardware.

---


# Conclusion

These assumptions provide a clear interpretation of the coding challenge requirements and ensure that the implementation behaves deterministically for all tested scenarios. Where the specification leaves behavior undefined, conservative security-oriented decisions were made to preserve control-flow integrity.
